## Défis

- **Story 2.5 — 2 bugs critiques rattrapés par le test end-to-end, pas par la review :**
  - Joueurs actifs sans résultat (nouveaux inscrits après Round 1) étaient **exclus** du classement suisse au lieu d'être ajoutés en fin.
  - Route `/validate` calculait le classement **global multi-journées** au lieu d'intra-journée → AC #5 violé.
  → Les deux fixes sont en place, mais c'est le test de Brice sur 3 journées qui les a révélés.
- **Story 2.6 — 6 décisions métier tranchées à chaud** — transitions d'état drop, un-drop, audit trail, visibilité des absents. Pas des bugs mais des trous dans le PRD qu'il a fallu combler au vol.
- **Incohérence de statut dans le fichier 2.6** — `Status: review` dans le markdown alors que sprint-status indiquait `done`. Corrigé pendant la rétro.
- **Rate limiter en mémoire a bloqué les tests API** (2.5) — détail mineur mais à garder en tête pour faciliter les tests manuels en dev.
- **Processus nodemon coincé sur port 3001** (2.5) — perte silencieuse de ~10 min de debug. Node exécuté directement via `node dist/index.js` a résolu.

## Enseignements clés

1. **Le test end-to-end est un filet de sécurité indispensable, pas un "nice to have"** — en 2.5, la code review a laissé passer 2 bugs critiques que seul un test réel a détectés. Pour Epic 3+, prévoir un test bout en bout par story, pas uniquement en fin d'epic.
2. **Un modèle de données bien pensé paie ses intérêts** — les champs `placement` et `points` ajoutés dans `LobbyPlayer` dès la 2.3 (en prévision de 2.4) et le filtre `status: 'inscrit'` ont permis à 2.6 de se limiter à une ligne de code backend.
3. **Les services purs testables sont le meilleur investissement de cette epic** — 3 services (`lobbyGenerator`, `pointsCalculator`, `swissSystem`), 30 tests, 0 régression. Pattern à continuer pour Epic 3 (tri classement, calcul progression victoire Epic 5).
4. **Le PRD n'est pas figé — il évolue avec les décisions produit** — la suppression de la limite 6 rounds (validée avec Brice) montre que le plan initial doit être réconcilié avec la réalité d'implémentation. À refaire systématiquement.
5. **La status hygiene compte** — le décalage entre `Status:` dans le .md et sprint-status (cas de 2.6) casse la traçabilité. À harmoniser lors de la passe de review.

## Découverte significative — Suppression de la limite de rounds

**Décision prise par Brice pendant 2.7, validée le 2026-04-17 :**
- La constante `MAX_ROUNDS = 6` a été retirée du backend.
- La journée se termine désormais via un bouton explicite "Terminer la journée" dans le backoffice.
- Raison : flexibilité — selon le déroulé réel du tournoi, le nombre de rounds optimal peut varier. Pas besoin d'une contrainte dure.

**Impact sur le plan :**
- **PRD FR17** mis à jour : "nombre illimité de rounds, fin manuelle" — fait pendant la rétro.
- **epics.md** — mapping FR17 + Story 2.5 AC#4 mis à jour pour refléter la fin manuelle.
- **Epic 3** : impact nul — la page publique affiche le classement peu importe le nombre de rounds.
- **Epic 5 (finale)** : cohérent avec la condition de victoire "top 1 + ≥20 pts cumulés" qui est déjà une condition dynamique, pas un compte de rounds.
- **Mémoire** : `project_rounds_per_day.md` déjà existante, conservée comme référence.
