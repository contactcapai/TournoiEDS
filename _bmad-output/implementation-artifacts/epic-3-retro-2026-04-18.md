## Défis

- **Bug multi-journées sur Story 3.2 détecté par Brice, pas par la review LLM externe** — la première version indexait les résultats sur `roundNumber` seul ; or le schéma Prisma a une contrainte `@@unique([dayId, number])` qui réinitialise `roundNumber=1` à chaque nouvelle journée. Conséquence : `find((r) => r.roundNumber === 1)` matchait J1R1 en premier et les colonnes J2 étaient invisibles dans le tableau. Le LLM externe avait pourtant approuvé la story. **Le test E2E de Brice sur J1+J2 en local a rattrapé** — validation directe de l'action item "test E2E par story" de la rétro Epic 2.
  - Fix v1.1 : ajout de `dayNumber` dans `PlayerRoundSummary`, header tableau 3 niveaux ("Journée N" / "R1/R2" / "Place/Pts"), lookup de cellule par `(dayNumber, roundNumber)`, bordure dorée entre journées.

- **Quirks lint React 19** — deux règles nouvelles ont obligé des adaptations :
  - `react-hooks/refs` : accès `ref.current` au render interdit → suppression de `hasRevealedRef` (remplacé par le comportement natif `initial → animate` au mount des `<motion.tr>`).
  - `react-hooks/set-state-in-effect` : setState synchrone dans le body d'un `useEffect` interdit → `setFlashingIds` différé via `setTimeout(fn, 0)`.
  - Fonctionnellement identique, mais à documenter comme pattern pour Epic 4+.

- **Flash update invisible sur cellules sticky** — les colonnes Rang/Pseudo/Total ont un fond opaque alterné (sticky left-0) qui masquait un `animate-[rankingFlash]` posé sur `<tr>`. Solution : appliquer la classe sur chaque `<td>` individuellement.

- **Partenaires repositionnés 3 fois sur Story 3.3** — PartnersSection bas de page → PartnersMarquee dans le hero (desktop row / mobile stack) → PartnersMarquee juste sous le tableau (final). Signal que le scope produit n'était pas figé en début de story. Pas un bug, plutôt une itération UX naturelle.

- **Logos partenaires lourds** — 4 PNG non optimisés : 10 + 18 + 177 + 261 kB = **~467 kB bruts**. Les 2 plus lourds (LOGO-V3-BLANC 177 kB et clavicule 261 kB) gagneraient fortement à être convertis en WebP ou compressés. Chargés en `loading="lazy"` donc FCP pas impacté, mais Lighthouse mobile en souffrira.

- **Pas de Status review sur 3.2** — ce que Brice avait demandé en Epic 2 est tenu, mais noter que la review LLM externe n'a **pas** intercepté le bug multi-journées qui a été rattrapé en session avec Brice (v1.1 directement après la v1.0). L'étape "review" reste utile mais n'est pas suffisante à elle seule — le test E2E reste le filet critique.

## Enseignements clés

1. **Le test E2E par story est validé comme filet critique** — en Story 3.2, la review LLM externe a approuvé une implémentation avec un bug multi-journées. Seule la validation navigateur de Brice sur J1+J2 l'a révélé. **L'action item de la rétro Epic 2 a concrètement sauvé la prod.**

2. **`dayNumber` obligatoire dans toute agrégation multi-journées** — règle à ancrer : dès qu'un identifiant métier peut être réinitialisé par un parent (ici `Round.number` par `Day`), il faut **toujours** le combiner avec l'identifiant du parent dans les clés/lookups côté frontend. À garder en mémoire pour Epic 5 (si les rounds de finale réinitialisent aussi leur numérotation).

3. **Framer-motion + React 19 a des contraintes lint spécifiques** — pas de `ref.current` lu au render, pas de setState synchrone dans `useEffect` body. Les patterns workaround (`setTimeout(0)`, variants passés en permanence) sont propres une fois connus. À documenter dans Previous Story Intelligence de Story 4.1.

4. **Cellules sticky masquent les animations `background-color` sur `<tr>`** — pour toute animation visuelle sur une ligne avec sticky cells, appliquer la classe sur chaque `<td>` plutôt que sur `<tr>`.

5. **Bundle framer-motion raisonnable** — +41 kB gz pour avoir `layout`, `staggerChildren`, `useReducedMotion`, `AnimatePresence`. Rentable. Pas besoin de fallback CSS-only prévu dans la story.

6. **Le namespace Socket.IO unique `/tournament` est le bon choix** — pensé dès 3.1 pour servir Epic 3 (public), Epic 4 (overlay) et Epic 5 (finale). Aucune refacto à prévoir, réutilisation pure.
