## Défis — 3 bugs critiques découverts **après** le "done" dev

**Point structurel majeur** : les 3 bugs ci-dessous ont tous été trouvés par Brice en validation manuelle e2e locale, **pas** par les tests automatisés. Brice confirme avoir eu besoin de **plusieurs itérations de tests** pour les détecter. C'est une limite structurelle acceptée — les tests unitaires ne couvrent pas les flux multi-journées finale, et le projet n'a délibérément pas de framework de tests d'intégration HTTP (décision rétro Epic 2 reconduite).

### Bug #1 — `computeLobbyGroups` : Swiss multi-journées cassé (dette Epic 2)

- **Origine** : stories 2-3 / 2-5 (Epic 2)
- **Découvert** : validation e2e 5.2, alors que Brice testait une finale avec 3 journées qualif complètes
- **Nature** : J1R1 devait être la seule génération aléatoire de lobbies ; pour tous les autres rounds (J2R1, J3R1, etc.), la logique Swiss devait utiliser `aggregateQualificationRankings()`. En pratique, la logique était incorrecte et produisait des lobbies non conformes au Swiss multi-journées.
- **Correctif** : hotfix 5.2 v1.2 — `computeLobbyGroups` corrigée, documentation mise à jour (`README.md` + `docs/CONTEXTE-PROJET.md`)
- **Enseignement** : un bug métier peut dormir pendant 3 épics avant d'être détecté si aucun scénario e2e ne l'exerce. Les tests unitaires `lobbyGenerator` / `swissSystem` testaient les bonnes primitives, mais l'orchestration `computeLobbyGroups` n'était pas couverte par un test multi-journées.

### Bug #2 — Bouton "Régénérer lobbies" supprimait le lobby unique finale

- **Origine** : oubli 5.1 (garde manquante)
- **Découvert** : validation e2e 5.2
- **Nature** : `POST /generate-lobbies` et `POST /regenerate-lobbies` n'avaient aucune garde pour refuser l'exécution si la Day en cours est de type `finale`. Cliquer "Régénérer" en finale supprimait le lobby unique de 8 finalistes et recréait des lobbies Swiss depuis tous les inscrits.
- **Correctif** : hotfix 5.2 v1.2 — garde `FINALE_LOBBY_IS_FIXED` ajoutée côté backend + bouton masqué côté frontend en finale
- **Enseignement** : un invariant métier ("le lobby finale est immuable") doit être explicitement codé en garde backend **ET** explicitement masqué côté UI. Ne pas compter sur "l'admin ne cliquera pas là". Le principe "Defense in Depth" à retenir.

### Bug #3 — Règle de victoire mal implémentée (spec ambiguë)

- **Origine** : ambiguïté AC #3 de 5.2 ("≥ 20 pts cumulés" sans préciser avant/après le round)
- **Découvert** : revue manuelle de **5.3** par Brice (pas 5.2 !)
- **Nature** : `detectFinaleWinner` vérifiait `totalScore >= 20` (**post-round**). La règle réelle, clarifiée par Brice pendant la revue : le seuil doit être atteint **avant** le round Top 1 (cumul des rounds antérieurs). Traverser 20 pts PENDANT le round du Top 1 ne doit PAS déclencher la victoire.
- **Correctif** : hotfix 5.2 v1.4 — `detectFinaleWinner` réécrit en `preRoundTotal >= 20` (on soustrait les points du dernier round). Tests réécrits avec `roundResults` réalistes + 4 nouveaux cas limites : crossing 14+8=22 → null, crossing 19+8=27 → null, `roundResults = []` → null, ordre non trié. 60 → **64 tests backend verts**.
- **Impact en cascade** : 5.3 v1.1 — badge `✓ Top 1` (basé sur `top1Count >= 1`, trompeur) retiré du frontend, remplacé par `⚡ Éligible victoire` (affiché lorsque `totalScore >= 20`) avec tooltip explicite. Alignement admin `DayManager` en 5.3 v1.2 (badge `✓ Top 1` → `⚡ Éligible`).
- **Enseignement majeur** : un service pur avec 8 tests verts peut tester la **mauvaise règle**. Les tests confirment l'implémentation, pas la correspondance à la règle métier réelle. L'AC #3 disait "≥ 20 pts cumulés" — syntaxiquement correct, sémantiquement ambigu. **La clarification est venue de Brice, pas du dev.** Position Brice (acté en rétro) : pas de nouvelle règle de process à ajouter (pas de relecture produit obligatoire, pas d'exemples numériques forcés dans les AC) — on accepte que ce type de bug se détecte en itération e2e.

### Autres défis mineurs

## Enseignements clés

1. **Les bugs métier latents remontent en validation e2e, pas en dev** — les 3 bugs critiques sont tous apparus pendant les tests manuels de Brice, pas pendant le dev. Position acceptée : pas de process supplémentaire à imposer (pas de tests d'intégration HTTP forcés, pas de relecture produit formelle). La validation e2e itérative de Brice **est** le filet de sécurité. Cela suppose que Brice garde la discipline de tester chaque story en local avant de la passer `done`.

2. **Un service pur + tests verts ne garantit pas la correspondance à la règle métier** — `winnerDetector` en est la preuve. Les tests vérifiaient que le code faisait ce que la spec disait ; la spec était ambiguë ; donc les tests étaient verts sur la mauvaise règle. Le garde-fou c'est la validation e2e par le Project Lead qui porte la règle métier en tête. Pas de workaround process à ce stade.

3. **Un invariant métier doit être gardé côté backend ET côté UI** — bug #2 (lobby finale fixe) aurait pu être évité par une simple garde backend. Règle à retenir pour Epic 6 et au-delà : tout invariant "cet état est immuable/interdit" = garde backend explicite (avec code d'erreur dédié) + masquage UI. Jamais compter sur "l'admin n'ira pas là".

4. **La réutilisation Epic 3–4 a tenu ses promesses sur 5.3** — `useRankingFlash` intact, pattern `motion.tr layout` + `layoutId` préfixé, tokens Tailwind EDS existants. Résultat : +1.04 kB gz pour toute une page publique. **Pattern à reconduire** : quand l'extrait de composant / hook est stable et testé sur une page, le réutiliser tel quel est toujours plus sûr que tenter de mutualiser via prop `variant` (décision prise en Epic 4 sur `OverlayRankingTable` vs `FinaleRankingTable` séparés).

5. **Le Change Log granulaire d'une story (v1.0 → v1.4) est un pattern précieux** — 5.2 a 5 versions dans son Change Log, chacune datée et attribuée. On peut reconstituer la chronologie exacte sans dépendre de `git log`. Pattern à maintenir pour toutes les stories avec hotfixes post-review.
